## Context and Problem Statement

JPA entity classes require a no-arg constructor, and getters and setters for every
persistent field. Writing this by hand is mechanical, verbose, and error-prone at scale.
Several code-generation approaches exist, each with different tradeoffs for JPA
compatibility and test-coverage tool accuracy.

## Decision Drivers

- JPA requires a public or protected no-arg constructor and field accessors on every entity;
  hand-writing these for every field is error-prone and obscures the domain model in diffs.
- Lombok's `@Data` and `@EqualsAndHashCode` are unsafe on JPA entities: `@Data` generates
  `equals`/`hashCode` over all fields including mutable ones and can trigger lazy-load
  traversals; `@EqualsAndHashCode` on a bidirectional association causes infinite recursion.
- JaCoCo counts branches in generated bytecode unless the generator marks its methods with
  `@Generated`; unmarked generated code pollutes coverage reports and makes the 80% gate
  harder to interpret accurately.
- Java records provide immutable value semantics ‚Äî `equals`, `hashCode`, `toString`, and a
  canonical accessor per component ‚Äî out of the box, so DTOs do not need Lombok at all.

## Considered Options

- Lombok `@Getter` / `@Setter` / `@NoArgsConstructor` on entities only; DTOs as Java
  records; `lombok.config` marks generated methods for JaCoCo exclusion
- Lombok `@Data` on entities (unsafe for JPA ‚Äî generates mutable `equals`/`hashCode`)
- Hand-written boilerplate (too verbose at scale; pollutes diffs with non-domain noise)
- IDE-generated boilerplate committed to source (must be re-generated on every field change;
  pollutes every entity diff)

## Decision Outcome

Chosen option: **Lombok `@Getter`, `@Setter`, and `@NoArgsConstructor` on JPA entity
classes; DTOs as Java records; `lombok.config` with `addLombokGeneratedAnnotation =
true`**, because it eliminates entity boilerplate safely without the JPA pitfalls of
`@Data`, and keeps JaCoCo coverage numbers meaningful by excluding generated accessors.
Lombok is not confined to entities ‚Äî it may also reduce boilerplate elsewhere (see rule 5).

The rules are:

1. **Permitted on JPA entities:** `@Getter`, `@Setter`, `@NoArgsConstructor`. These three
   annotations are the only Lombok annotations used on entity classes.
2. **Forbidden on JPA entities:** `@Data` and `@EqualsAndHashCode`. Both are prohibited
   because they generate `equals`/`hashCode` semantics that are unsafe on managed JPA
   entities (mutable-field hashing, potential lazy-load traversal, infinite recursion on
   bidirectional associations).
3. **DTOs are Java records, not Lombok classes.** Records give immutable value types with
   `equals`, `hashCode`, `toString`, and accessors generated by the compiler. Lombok is not
   used on DTO types.
4. **`lombok.config` at the project root sets `lombok.addLombokGeneratedAnnotation = true`**
   so that Lombok annotates every generated method and constructor with `@Generated`. JaCoCo
   respects this annotation and excludes those methods from branch and line coverage
   counting, keeping the 80% gate focused on real logic.
5. **Lombok beyond entities is allowed where it removes boilerplate.** Lombok may be used on
   any class where it eliminates boilerplate without introducing a drawback ‚Äî for example
   `@RequiredArgsConstructor` on a service to generate constructor injection. The `@Data` and
   `@EqualsAndHashCode` prohibition above is specific to the hazards of *JPA entities* and
   does not extend to other types; outside entities, judgement is the only gate.

### Consequences

- Good: Entity boilerplate is eliminated without compromising JPA semantics or triggering
  the unsafe behaviours of `@Data`.
- Good: The `addLombokGeneratedAnnotation` setting keeps JaCoCo coverage numbers meaningful
  ‚Äî generated accessors and constructors are excluded, so the 80% gate measures real logic.
- Good: Java records for DTOs provide correct value semantics (immutable, `equals`,
  `hashCode`, `toString`) with zero extra dependencies.
- Good: The forbidden-annotation rule (`@Data`, `@EqualsAndHashCode` on entities) is
  explicit and can be enforced via code review or a Checkstyle rule.
- Bad: Lombok is an annotation processor; misconfiguration or a version mismatch with the
  Java or Spring Boot version can produce cryptic compilation errors.
- Bad: The prohibition on `@Data` and `@EqualsAndHashCode` on entities is a project rule,
  not a compiler error ‚Äî developers must learn and apply it deliberately.
