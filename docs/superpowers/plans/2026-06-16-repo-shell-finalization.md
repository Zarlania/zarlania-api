# Repo Shell Finalization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the remaining gaps between the approved repo-shell spec
(`docs/superpowers/specs/2026-06-15-zarlania-api-repo-shell-design.md`) and the live
implementation: enable the spec'd health probes, drop the leftover POC endpoint, write the
missing governance ADRs, accept the proposed ones, clarify the ADR trigger for endpoints,
and revamp the human/AI entry docs to match reality.

**Architecture:** Three workstreams in order — (A) small app-shell code corrections, (B)
ADR governance backfill + acceptance, (C) docs revamp — then a final spec-vs-implementation
review whose findings seed the PR description. The permanent record of each divergence lives
in its ADR; the working registry file is **removed before merge** so it never lands on `master`.

**Tech Stack:** Java 25 / Spring Boot 4.1.x / Maven (`./mvnw`), Spring Boot Actuator,
springdoc-openapi, the `./scripts/adr` Python CLI, pre-commit, GitHub Actions.

## Workflow & ground rules (read before starting)

- **Issue-driven, per `CONTRIBUTING.md`.** Every PR references an issue (`#<n>` in title or
  `type/<issue#>-slug` branch). Group the tasks into PRs as noted below; open a GitHub issue
  for each PR that doesn't already have one. Known issue: **#22** (app-shell deploy) covers
  Workstream A. Render Blueprint migration (spec §7) is **already tracked as a separate
  GitHub issue — out of scope here.**
- **Release label:** each PR carries a `release:*` label and bumps `pom.xml` via
  `./scripts/bump-version bump <kind>`. Suggested: Workstream A = `release:patch`
  (no public contract added — `/` is removed, probes added); B and C = `release:patch`
  (docs/governance only). Confirm with the maintainer at PR time.
- **`docs/superpowers/**` is frozen** — exempt from linters and review edits. Do **not**
  edit the spec. Divergences are recorded in ADRs and the registry in Task 12, not in the spec.
- **Validate locally before every commit:** `./mvnw -q verify` for Java, `./scripts/adr check`
  for ADRs, `pre-commit run --all-files` for lint/hygiene.

```markdown
## Context and Problem Statement

Every change to a live public service must pass consistent format, lint, static-analysis,
security, and coverage gates — fast feedback locally, full enforcement in CI.

## Decision Drivers

- Fast pre-commit (no compilation/tests) so commits stay cheap.
- Authoritative, comprehensive gates in CI before merge/deploy.
- One coverage bar across languages; secrets can never land in a commit.

## Considered Options

- Maven-driven gates + pre-commit/CI split (chosen).
- Run everything in pre-commit (too slow).
- CI-only (loses fast local feedback).

## Decision Outcome

Chosen gates: **Spotless (google-java-format) + Checkstyle (google_checks)** run fast in
pre-commit and CI; **SpotBugs (effort Max) + FindSecBugs 1.14.0 + JaCoCo (line AND branch

hygiene hooks. CI posts a JaCoCo coverage comment to the PR.

(tested with pytest + coverage ≥ 80), not bash — so there are no bats/kcov tests**; and CI
gitleaks scans the **PR commit range** rather than full history, because the repository was
created with clean history and pre-commit blocks new secrets at staging time.

### Consequences

- Good: fast local loop, strong CI enforcement, coverage visible inline, secrets blocked
  at two layers.
- Bad: heavy gates only surface in CI (slower than local for those); per-language coverage
  config must be maintained as new script languages appear.

## Links

- Spec: `docs/superpowers/specs/2026-06-15-zarlania-api-repo-shell-design.md` §4
```

Body:

```markdown
## Context and Problem Statement

Every merge to `master` deploys once to production. We want each merge to also cut exactly
one SemVer release with no manual steps and no double deploy.

## Decision Drivers

- The merge commit must be the only push (Render deploys once) — no post-merge bump commit.
- Releasing must be automatic and hard to get wrong.
- No separately maintained changelog file.

## Considered Options

- In-PR version bump + automated tag/Release on merge (chosen).
- Post-merge bump commit (causes a second deploy — rejected).
- Manual tagging/releases (error-prone — rejected).

## Decision Outcome

Chosen: **SemVer lives in `pom.xml <version>`** (surfaced at `/actuator/info`). A
`release:major|minor|patch` label selects the bump (unlabeled = patch); the bump is
committed **inside the PR** (`./scripts/bump-version`). A CI "Release version bump" check
validates the in-PR version against the label relative to the latest release tag. On merge,
`release.yml` creates the `v<version>` tag and a GitHub Release with auto-generated notes (no
`CHANGELOG.md`). Dependabot PRs are exempt and ride the next human release.

### Consequences

- Good: one merge = one deploy = one release; releasing is automatic and validated.
- Bad: contributors must remember the label/bump (mitigated by the CI check and PR template);
  release notes quality depends on PR titles.

## Links

- Spec: `docs/superpowers/specs/2026-06-15-zarlania-api-repo-shell-design.md` §6
- `pom.xml`, `scripts/bump-version`, `scripts/release_tool/`, `.github/workflows/release.yml`
```
